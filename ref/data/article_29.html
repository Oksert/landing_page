Сообщения администратору об ошибках можно разделить на 2 типа:

  • Ошибки, возникающие при авторизации – к такому типу ошибок относятся ошибки при вводе логина и/или пароля, в этом случае на странице логина выводится соответствующее сообщение.

  • Ошибки, возникающие в ходе работы программного комплекса – к такому типу ошибок относятся сообщения, возникающие непосредственно в ходе работы с системой. Примерами таких ошибок могут следующие виды сообщений системы:
    • В случае одновременной работы двух или более пользователей с одной записью при попытке внесения изменений будет выведено сообщение вида: «2015-07-01 18:00:09: Обновляемая запись была обновлена Ловков Константин с момента считывания». В этом случае нужно дождаться, когда соответствующий пользователь закончит работу с данной записью, закрыть и открыть её, затем снова внести все необходимые изменения.
    • В случае, если открытая запись была удалена будет выведено сообщение следующего вида: «2015-06-09 16:47:05: record is NULL. Unable to process rules» или «2015-07-02 10:47:23: Обновляемая запись была удалена с момента считывания.».
    • В случае наличия ошибок в SQL-коде, используемом во время работы системы будет выведено сообщение вида «2015-06-08 12:19:08: You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version for the right syntax to use near '…'» или «2015-06-09 11:51:04: java.sql.SQLException: java.lang.NullPointerException». В этом случае необходимо внести исправления в соответствующий SQL-код. Определить в каком именно SQL-коде ошибка можно в логах сервера – поиск при этом удобнее осуществлять по времени сообщения.
    • В случае отсутствия таблицы или представления, необходимых для работы системы будет выведено сообщение вида «2015-06-23 14:42:48: Table 'rpm_prod.v_form_1_2' doesn't exist». В таком случае необходимо создать соответствующую таблицу или преставление исходя из логики использования её системой.
    • Если при попытке создания индекса, который будет использоваться в качестве первичного ключа реконсиляции, в таблице, для которого создается такой индекс, будут существовать дублирующие значения по такому ключу, то будет выведено сообщение вида «2015-06-15 16:42:37: Невозможно создать уникальный ключ. Таблица 'sm_tasktemplate' содержит 847 дублирующих значений». В этом случае необходимо либо изменить состав полей индекса, либо удалить дубли записей.
    • Если при попытке использования ClassLibrary необходимый метод отсутствует, то будет выведено сообщение вида «2015-06-11 10:34:43: Static method enableSupportTaskAutoTransition(ru.ip.server.entity.EntityDTO) not found in class'ru.classlibrary.Lib$SDUtils'». В данном примере в библиотеке «SDUtils» отсутствует метод «enableSupportTaskAutoTransition». Необходимо либо исключить использование этого метода в правилах, либо добавить соответствующий метод в библиотеку.
    • Если при попытке использования ClassLibrary система выдает сообщение вида «2015-06-24 13:54:53: Class or variable not found: Lib.RequestUtils». В данном примере отсутствует класс «RequestUtils». Необходимо либо создать его и перезагрузить сервер, если он отсутствует в системе, либо перезагрузить сервер, если он существует.
    • Если при попытке создания новой записи не заполнено обязательное поле, то будет выведено сообщение вида «2015-06-30 11:40:24: Поле Внутреннее имя не заполнено». Заполните необходимое поле (в приведенном примере «Внутреннее имя») и нажмите «Создать».
    • Если при попытке создания новой записи нарушается условие уникальности индекса, то будет выведено сообщение вида «2015-07-01 11:30:58: Нарушено Ограничение 144. Значение поля DISPLAYNAME должно быть уникальным для таблицы 'Формат Поля'». В этом случае необходимо изменить данные во вносимой записи исходя из условий индекса.
    • Если при открытии отчета появляется сообщение вида «2015-06-29 17:19:21: Subquery returns more than 1 row», то необходимо исправить код запроса отчета (или код представлений, используемых в коде запроса отчета) так, чтобы подзапросы не возвращали более одной строки.
